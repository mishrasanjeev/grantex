AI Agent Authorization Explained

How delegated authorization works for autonomous AI agents — the concepts, the protocol, and the code

Why AI Agents Need Authorization

AI agents are no longer just answering questions. They book flights, process payments, query databases, send emails, and manage infrastructure. They operate autonomously, often chaining together multiple services in a single workflow.

This creates a fundamental security problem: how do you give an AI agent access to real services without giving it unrestricted access?

Traditional approaches fall short. API keys grant all-or-nothing access with no identity, audit trail, or revocation granularity. OAuth 2.0 was designed for human users granting access to applications, not for autonomous agents that spawn sub-agents, delegate tasks, and operate without continuous human oversight.

AI agent authorization solves this by introducing a delegated authorization model built specifically for agents. The core idea: a human explicitly approves a set of scoped permissions for an agent, the agent receives a cryptographically signed token encoding exactly those permissions, and any downstream service can verify that token independently.

This is the model that Grantex implements as an open protocol. Think of it as OAuth 2.0 rebuilt from the ground up for agentic workloads, with first-class support for agent identity, delegation chains, hash-chained audit trails, real-time revocation, and budget controls.

How Agent Authorization Works

The agent authorization protocol follows a five-step flow. If you have worked with OAuth 2.0, the shape will be familiar, but each step carries agent-specific semantics.

1
Authorization Request The agent (or its host application) sends a POST /v1/authorize request specifying the agent ID, the target scopes, and the principal (human user) who should approve. The server returns an authRequestId, a consentUrl, and an expiration time.
|
2
Human Consent The principal visits the consent URL and sees exactly what scopes the agent is requesting. They can approve, deny, or narrow the scopes. This step is critical: no agent gets permissions without explicit human approval.
|
3
Token Exchange After approval, the application exchanges the authorization code for a grant token via POST /v1/token. The response includes a signed RS256 JWT (grantToken), the approved scopes, a refresh token, a grant ID, and the token's expiration.
|
4
Token Verification Any service that receives the grant token can verify it. Offline verification uses the issuer's public JWKS endpoint to validate the RS256 signature without any network call to the auth server. Alternatively, services can call POST /v1/tokens/verify for online verification that also checks revocation status.
|
5
Revocation When an agent's access should end, call POST /v1/tokens/revoke. Revocation takes effect in under one second globally. If the revoked grant has delegated child grants, all children are automatically cascade-revoked.

This flow ensures that every agent action is traceable to a human approval, every token is cryptographically verifiable, and every permission can be revoked instantly.

Understanding Grant Tokens

The grant token is the central artifact of AI agent authorization. It is a signed RS256 JWT that encodes everything a service needs to make an authorization decision: who approved the access, which agent is acting, what scopes are permitted, and when the token expires.

JWT Claims

Grant Token Payload
issIssuer -- the auth server URL
subPrincipal ID -- the human who approved this grant
agtAgent DID -- cryptographic identity of the agent
devDeveloper ID -- the developer who registered the agent
scpScopes array -- the permissions this agent was granted, e.g. ["email:read", "calendar:write"]
grntGrant ID -- unique identifier for this authorization grant
jtiToken ID -- unique identifier for this specific JWT
iatIssued at -- Unix timestamp of when the token was created
expExpiration -- Unix timestamp after which the token is invalid

The key difference from standard OAuth JWTs is the agt claim. This binds the token to a specific agent identity, not just a client application. Combined with sub (the approving human), every token answers two questions: who approved this? and which agent is acting?

Offline Verification

Services verify grant tokens by fetching the issuer's JWKS (JSON Web Key Set) and validating the RS256 signature locally. This means verification requires zero network calls to the auth server after the initial JWKS fetch (which is cached). A service behind a firewall, in a different cloud region, or operating in an air-gapped environment can all verify tokens independently.

Agent Identity (DID)

In traditional authorization systems, the identity boundary is the application. OAuth issues tokens to a client ID representing an app. But a single application can run many agents -- a travel planner, a flight booker, a hotel finder, a budget tracker. Under OAuth, they all share the same client credentials. There is no way to distinguish which agent performed an action, scope permissions per agent, or revoke one agent without affecting the others.

Grantex solves this with Decentralized Identifiers (DIDs). Every registered agent receives its own DID, a globally unique cryptographic identifier that is embedded in the agt claim of every grant token the agent receives.

This has several concrete consequences:

  • Per-agent scoping -- different agents under the same developer can have different permission sets
  • Per-agent revocation -- revoke one agent without touching others
  • Per-agent audit -- every audit log entry includes the acting agent's DID
  • Per-agent budgets -- spending limits are bound to individual agents, not shared across an application
  • Verifiable identity -- the DID can be resolved to a DID Document containing the agent's public key, enabling cryptographic proof of agent identity independent of the auth server

Agent identity is what makes the difference between "some API client made this request" and "the flight-booking agent (did:web:grantex.dev:agents:flight-booker) made this request on behalf of Alice, with her approval, using the email:read scope."

Delegation Chains

Real-world agentic workflows are rarely flat. A manager agent assigns a research task to a researcher agent, which delegates data retrieval to a specialized crawler agent. This creates a chain of delegation that traditional auth systems cannot represent.

Grantex supports multi-level delegation natively. A parent agent can delegate a subset of its scopes to a child agent. The child receives its own grant token with additional claims that track the delegation:

  • parentAgt -- the DID of the parent agent that delegated access
  • parentGrnt -- the grant ID of the parent's authorization
  • delegationDepth -- an integer tracking how many levels deep this delegation is (0 = root, 1 = first delegate, etc.)

Two critical rules enforce security in delegation chains:

  1. Scope narrowing -- a child can only receive a subset of its parent's scopes, never additional ones. If the parent has ["flights:book", "hotels:book"], it can delegate ["flights:book"] to a child, but it cannot grant the child ["payments:process"].
  2. Cascade revocation -- revoking a parent's grant automatically and immediately revokes all of its children's grants, recursively. This ensures that cutting off a rogue parent agent instantly disables its entire sub-agent tree.
// Parent agent delegates a narrower scope to a child agent
const delegated = await gx.grants.delegate({
  parentGrantToken: parentToken,
  childAgentId: flightBooker.id,
  scopes: ['flights:book'],  // subset of parent's scopes
});

// The child's JWT includes delegation tracking:
// { parentAgt: "did:web:...:travel-planner",
//   parentGrnt: "grnt_abc123",
//   delegationDepth: 1 }

The Authorization Lifecycle

Agent authorization is not a one-time event. It is a lifecycle with four phases:

Create

A human approves scoped permissions for an agent. The agent receives a grant token. This is the consent moment -- the human sees exactly what the agent will be able to do and explicitly approves it.

Use

The agent presents its grant token to downstream services. Each service verifies the token's signature, checks the scopes, and allows or denies the request. The agent can refresh its token using the refresh token endpoint (POST /v1/token/refresh) to get a new JWT without requiring the human to re-approve.

Audit

Every action the agent performs is logged in a hash-chained, append-only audit trail. Each entry includes the agent DID, grant ID, action performed, scopes used, a timestamp, and a SHA-256 hash linking to the previous entry. This makes the log tamper-evident: altering any past entry breaks the hash chain.

Principals can review their agents' activity via the principal dashboard, and developers can query the audit API programmatically.

Revoke

When access should end -- the task is complete, the agent misbehaved, or the user simply changed their mind -- the grant is revoked. Revocation propagates globally in under one second. Cascade revocation ensures that revoking a parent kills the entire delegation tree. There is no window where a revoked agent can still operate.

Getting Started

Grantex is open source (Apache 2.0) with SDKs for TypeScript, Python, and Go. Here is the complete authorization flow in both TypeScript and Python.

import { GrantexClient } from '@grantex/sdk';

const gx = new GrantexClient({
  baseUrl: 'https://your-auth-server.com',
  apiKey: 'gx_live_...',
});

// 1. Register an agent
const agent = await gx.agents.register({
  name: 'travel-planner',
  scopes: ['flights:search', 'flights:book', 'email:read'],
});

// 2. Create an authorization request
const auth = await gx.authorize({
  agentId: agent.id,
  userId: 'user_alice',
  scopes: ['flights:search', 'flights:book'],
});

// 3. Direct the user to the consent URL
console.log('Approve at:', auth.consentUrl);

// 4. After approval, exchange the code for a grant token
const grant = await gx.tokens.exchange({
  code: authorizationCode,
  agentId: agent.id,
});

// grant.grantToken  -- signed RS256 JWT
// grant.scopes      -- ["flights:search", "flights:book"]
// grant.grantId     -- "grnt_..."
// grant.expiresAt   -- ISO 8601 timestamp

// 5. Verify the token (offline via JWKS or online)
const result = await gx.tokens.verify(grant.grantToken);
// result.valid, result.scopes, result.agent, result.principal

// 6. Revoke when done
await gx.tokens.revoke(grant.grantId);

Frequently Asked Questions

What is AI agent authorization?

AI agent authorization is the process of granting an autonomous AI agent specific, scoped permissions to act on a human's behalf. Unlike API keys, agent authorization uses signed JWTs with explicit scopes, cryptographic agent identity (DID), and human consent. This ensures you always know which agent did what, with whose approval, and you can revoke access in under one second.

How does delegated authorization work for AI agents?

Delegated authorization for AI agents follows a consent-based flow: the agent requests specific permissions, a human approves them via a consent URL, and the agent receives a signed JWT (grant token) with only the approved scopes. The agent presents this token when accessing services, and any service can verify it offline using JWKS. This is similar to OAuth 2.0 but adds agent identity, delegation chains, audit trails, and real-time revocation.

What is a grant token?

A grant token is a signed RS256 JWT issued after a human approves an agent's authorization request. It contains claims including the principal (human) ID, agent DID, developer ID, approved scopes, grant ID, and expiration time. Any service can verify the token offline using the issuer's public JWKS endpoint, requiring zero network calls to the auth server.

Can AI agents authorize other agents?

Yes. Grantex supports delegation chains where a parent agent can delegate a narrower subset of its scopes to a child agent. The child's JWT includes delegation claims (parentAgt, parentGrnt, delegationDepth) that track the chain. Children can never exceed their parent's permissions, and revoking the parent's grant automatically cascades to all delegated sub-agents.

How is agent authorization different from API keys?

API keys grant all-or-nothing access with no identity, scoping, consent, or audit trail. Agent authorization through Grantex provides per-agent scoped permissions, cryptographic identity for each agent, explicit human consent, hash-chained audit logs, real-time revocation, delegation control, and budget limits. For a detailed comparison, see Grantex vs API Keys.
