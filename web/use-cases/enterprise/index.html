Enterprise AI Agent Security

Governance, compliance, and authorization for AI agents at scale

Organizations are deploying AI agents across every function — customer support, code generation, data analysis, procurement, HR workflows, and financial operations. These agents act autonomously, make API calls on behalf of employees, access sensitive data, and interact with third-party services. The security implications are profound: every agent is an identity, every action is a decision, and every decision needs accountability.

Grantex is the open delegated authorization protocol built specifically for this problem. Where OAuth 2.0 was designed for users delegating to applications, Grantex is designed for principals — human or machine — delegating authority to AI agents. It provides the full enterprise security stack: SOC 2 compliance, SCIM provisioning, SSO, cryptographic identity, tamper-evident audit trails, a policy engine, and fine-grained delegation controls. All open-source under Apache 2.0, with an IETF Internet-Draft formalizing the protocol.

The Enterprise Challenge

Enterprise AI adoption is accelerating faster than the security infrastructure to support it. Teams across the organization are spinning up agents independently — engineering uses coding assistants with repository access, finance deploys agents that query billing APIs, marketing runs content generation pipelines that publish to CMS platforms. Without a centralized authorization layer, this creates shadow AI: agents operating with over-provisioned credentials, no audit trail, and no mechanism for revocation.

The risks are concrete. An agent with broad API keys can exfiltrate data. A compromised agent credential can be used to escalate privileges. When an employee leaves, their agents may continue operating indefinitely. Without delegation chains, there is no way to trace which human authorized which agent action. Compliance teams cannot answer the fundamental question: who authorized this agent to do this, and when?

Traditional API key management and OAuth 2.0 were not designed for this. API keys are static, unscoped, and lack identity semantics. OAuth 2.0 models user-to-application delegation, not the multi-tier delegation chains that emerge when agents spawn sub-agents or when a department head grants authority that cascades to team-level agents. Enterprise AI agent security requires a purpose-built authorization protocol.

Compliance and Audit

Meeting regulatory and internal compliance requirements is non-negotiable for enterprise deployments. Grantex provides the infrastructure to satisfy auditors, compliance teams, and regulators with cryptographic proof rather than procedural assurance.

SOC 2 Type I Certified

Grantex has completed SOC 2 Type I certification, demonstrating that the platform's security controls meet the Trust Services Criteria for security, availability, and confidentiality. The authorization service, key management, audit infrastructure, and operational procedures have all been independently evaluated. For organizations building on Grantex, this means a shorter path to their own compliance certifications — the authorization layer is already attested.

Hash-Chained, Tamper-Evident Audit Trails

Every authorization event in Grantex — grant issuance, token exchange, verification, revocation, delegation, budget debit — produces an audit log entry. Each entry is cryptographically linked to the previous one via hash chaining, creating a tamper-evident sequence. If any entry is modified or deleted after the fact, the chain breaks and the tampering is detectable. This is not a best-effort log; it is a cryptographic commitment to the history of every authorization decision.

Audit entries include the full context: which principal authorized the action, which agent performed it, what scopes were involved, the delegation depth, timestamps, and the cryptographic hash linking to the previous entry. Entries can be queried by grant, agent, principal, time range, or event type through the GET /v1/audit/entries API.

Real-Time Event Streaming for SIEM Integration

For security operations centers that need real-time visibility, Grantex provides event streaming via both Server-Sent Events (SSE) and WebSocket connections. Authorization events are emitted as they occur, enabling direct integration with SIEM platforms. The event destination connectors support Datadog, Splunk, Amazon S3, Google BigQuery, and Apache Kafka out of the box — covering the most common enterprise observability stacks.

Anomaly Detection

Grantex includes built-in anomaly detection that monitors authorization patterns and flags deviations. Unusual token exchange volumes, unexpected scope requests, verification failures from new IP ranges, and other signals are surfaced automatically. This provides an early warning system for compromised credentials or misconfigured agents, supplementing whatever threat detection your SOC already operates.

Identity and Access Management

Enterprise AI agent security starts with identity. If you cannot reliably identify an agent, you cannot authorize it, audit it, or revoke it. Grantex integrates with enterprise identity infrastructure and extends it to cover AI agents as first-class identities.

SCIM 2.0 Provisioning

Grantex supports SCIM 2.0 for automated user and agent lifecycle management. When an employee is provisioned in your identity provider, their Grantex developer account is created automatically. When they are deprovisioned, their account is disabled and all associated grants are revoked. This eliminates the orphaned-credential problem that plagues manual agent management and ensures that agent authority is always tied to an active human identity.

SSO via SAML 2.0 and OpenID Connect

Developers authenticate to Grantex through your existing identity provider. Both SAML 2.0 and OpenID Connect are supported, covering Okta, Azure AD, Google Workspace, Ping Identity, OneLogin, and any standards-compliant IdP. This means no new passwords to manage, no separate MFA enrollment, and centralized session policy enforcement.

FIDO2/WebAuthn Passkeys

For high-security operations — particularly grant authorization and delegation — Grantex supports FIDO2/WebAuthn passkeys. Developers can register hardware security keys or platform authenticators and require passkey verification for sensitive actions. This provides phishing-resistant authentication at the authorization boundary: even if an attacker compromises a session, they cannot authorize new agent grants without the physical passkey.

Cryptographic Agent Identity via DIDs

Every agent in Grantex is identified by a Decentralized Identifier (DID). The platform implements did:web infrastructure, publishing DID documents at /.well-known/did.json with Ed25519 public keys. This means agent identity is cryptographically verifiable — any party can resolve an agent's DID and verify that a grant token was issued to that specific agent. There is no reliance on a central authority for identity verification.

Policy Engine

Authorization without policy enforcement is just permission management. Grantex includes a policy engine that lets enterprises define, deploy, and enforce organizational rules governing agent behavior.

OPA-Compatible Policy Bundles

Policies are written as OPA-compatible bundles, giving teams access to the full Rego policy language. You can express rules that go far beyond simple scope checks: require manager approval for budget allocations above a threshold, restrict delegation depth for certain agent classes, enforce geographic constraints on data-processing agents, or mandate specific credential types for agents accessing PII. Bundles are uploaded via POST /v1/policies/sync and take effect immediately.

Git-Based Policy Sync

For teams that manage policy as code, Grantex supports webhook-triggered policy sync from Git repositories. Push a change to your policy repo, and the updated bundle is automatically deployed to the authorization service. This integrates policy management into your existing CI/CD workflows, provides version history and code review for policy changes, and ensures that the running policy always matches what is in source control.

Fine-Grained Scope Management

Scopes in Grantex are not coarse-grained read/write toggles. They are structured, hierarchical permission descriptors that can express exactly what an agent is permitted to do. Combined with the policy engine, scopes enable rules like: "this agent may read customer records in the EMEA region but not modify them" or "this agent may generate code but not deploy it to production." Grant tokens carry their scopes as JWT claims, enabling offline verification without calling back to the authorization server.

Delegation and Control

Real-world enterprise AI deployments involve chains of authority. A department head authorizes a team-level orchestrator agent, which delegates specific tasks to specialized sub-agents. Grantex models this natively.

Multi-Tier Delegation Chains

Grantex supports multi-tier delegation as defined in the protocol specification (Section 9). An authorized agent can delegate a subset of its own authority to another agent via POST /v1/grants/delegate. The delegated grant token includes claims for the parent agent DID, parent grant ID, and delegation depth, creating an auditable chain of authority back to the original human principal. Delegation depth can be limited by policy, preventing unbounded delegation chains.

Cascade Revocation

When a grant is revoked, all downstream delegations are revoked automatically. If a compromised agent is detected at any point in the delegation chain, a single revocation call disables the entire subtree. This is critical for incident response: security teams can immediately contain a breach by revoking a single grant, with confidence that no child agents retain authority. The revocation propagates through the hash-chained audit trail, creating a complete forensic record.

Budget Controls Per Agent and Department

For agents that incur costs — calling paid APIs, consuming compute resources, purchasing services — Grantex provides budget controls. Budget allocations are created per grant via POST /v1/budget/allocate, and every cost-incurring action debits the budget via POST /v1/budget/debit. When a budget is exhausted, the agent receives a 402 INSUFFICIENT_BUDGET response and cannot proceed. Budget balances and transaction histories are queryable in real time, giving finance teams and department heads direct visibility into agent spending.

Custom Domains

Enterprises can serve the Grantex authorization endpoints from their own domain (e.g., auth.yourcompany.com) via the custom domains feature. DNS verification ensures domain ownership, and all authorization flows — consent pages, token endpoints, DID documents — are served under your brand. This is particularly important for enterprises that require all security infrastructure to operate under corporate domain control.

Self-Hosted (Apache 2.0)

Deploy in your own VPC, on-premises data center, or air-gapped environment. Full source access, no license fees, no phone-home telemetry. You control the data plane entirely.

Terraform Provider

Kubernetes-native deployment via Helm charts with configurable replicas, resource limits, and ingress. Docker images available for standalone or Docker Compose deployments.

Security Architecture

The cryptographic foundations of Grantex are designed to provide verifiable security guarantees, not just procedural ones.

W3C Verifiable Credentials

Grantex issues W3C Verifiable Credentials (VCs) as VC-JWTs, providing standards-compliant, portable, and cryptographically verifiable claims about agent identity and authorization. Credentials include StatusList2021 support for revocation checking. Any relying party can verify a credential offline using the issuer's public key, without calling back to the Grantex service. This enables zero-trust architectures where agent security does not depend on network connectivity to a central authority.

SD-JWT Selective Disclosure

For scenarios requiring privacy-preserving credential presentation, Grantex supports SD-JWT (Selective Disclosure JSON Web Tokens). An agent can present a credential that proves it has authorization for a specific scope without revealing its full identity or the complete list of its permissions. This is essential for inter-organizational agent interactions where least-privilege disclosure is a requirement.

DID:web Infrastructure

Agent and issuer identities are anchored to did:web Decentralized Identifiers. DID documents are published at well-known URLs and contain Ed25519 public keys for signature verification. This decouples identity verification from any single service, enabling cross-organizational trust establishment. An agent's DID is embedded in every grant token (agt claim), creating an unbroken chain from token to identity to public key.

Offline Verification via JWKS

Grant tokens can be verified entirely offline using the issuer's JSON Web Key Set (JWKS). Services that consume agent tokens do not need to call the Grantex authorization server on every request — they cache the JWKS and verify JWT signatures locally. This eliminates the authorization server as a latency bottleneck and single point of failure, which is critical for high-throughput enterprise deployments. The TypeScript, Python, and Go SDKs all support JWKS-based offline verification.

Frequently Asked Questions

Is Grantex SOC 2 compliant?

Yes. Grantex has completed SOC 2 Type I certification. The platform provides hash-chained, tamper-evident audit trails, anomaly detection, and real-time event streaming for SIEM integration — all designed to satisfy the Trust Services Criteria for security, availability, and confidentiality.

Does Grantex support SCIM provisioning and enterprise SSO?

Yes. Grantex includes full SCIM 2.0 provisioning support for automated user lifecycle management, plus SSO via both SAML 2.0 and OpenID Connect. This allows enterprises to manage agent identities and developer access through their existing identity provider — Okta, Azure AD, Google Workspace, Ping Identity, OneLogin, and others.

Can Grantex be self-hosted in our own infrastructure?

Yes. Grantex is fully open-source under the Apache 2.0 license and can be deployed in your own infrastructure. The project provides a Terraform provider (5 resources, 2 data sources), Helm charts for Kubernetes, Docker images, and deployment configurations for GCP, AWS, and on-premises environments. You retain complete control over your data and authorization plane.

How do the audit trails work?

Grantex produces hash-chained, tamper-evident audit logs for every authorization event — grant issuance, token exchange, verification, revocation, and delegation. Each entry is cryptographically linked to the previous one, making retroactive modification detectable. Logs can be streamed in real time via SSE or WebSocket to your SIEM, and exported to destinations including Datadog, Splunk, S3, BigQuery, and Kafka.

How does Grantex help with AI agent compliance and governance?

Grantex provides a comprehensive governance layer for AI agents: fine-grained scope management limits what each agent can do, an OPA-compatible policy engine enforces organizational rules, budget controls cap spending per agent or department, cascade revocation instantly disables compromised delegation chains, and W3C Verifiable Credentials provide cryptographic proof of agent identity and authorization. Together, these controls give compliance teams full visibility and enforcement over every AI agent operating in the organization.
